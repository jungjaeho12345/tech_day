# SPEC-AUTH-001 — 인수 기준 (Acceptance Criteria)

> Given-When-Then 시나리오. 모든 기준은 관찰 가능한 동작으로 검증한다.

## 시나리오 1 — 로그인 성공 → 세션 수립 (정상 경로)

- **Given** DB(User 테이블)에 `userId`, 올바른 `password`(해시 저장), `role`을 가진 활성 사용자가 존재한다
- **When** 그 사용자가 로그인 폼에 일치하는 `userId` + `password`를 제출한다
- **Then** 시스템은 인증을 성공으로 판정하고, 서버 측 세션을 생성하며, 세션 식별자를 담은 세션 쿠키를 발급한다
- **And** 성공 신호를 반환하여 클라이언트가 기사 작성 페이지로 이동할 수 있게 한다 (news.md 58)
- **And** 응답 어디에도 저장된 비밀번호 해시가 포함되지 않는다 (REQ-AUTH-LOGIN-004)
- 커버: REQ-AUTH-LOGIN-001/002, REQ-AUTH-SESS-001/002/003

## 시나리오 2 — 로그인 실패 → 사유 ALERT (비정상 경로)

- **Given** 로그인 폼이 표시되어 있다
- **When** 사용자가 존재하지 않는 `userId` 또는 틀린 `password`를 제출한다
- **Then** 시스템은 인증을 거부하고 **세션을 수립하지 않으며**, 실패 사유를 담은 실패 결과를 반환한다 (REQ-AUTH-LOGIN-003)
- **And** 클라이언트는 그 사유와 함께 ALERT 메시지를 띄울 수 있다 (news.md 59)
- **And** 세션 쿠키가 발급되지 않는다
- 커버: REQ-AUTH-LOGIN-003, REQ-AUTH-SESS-001(미발생)

## 시나리오 3 — DPS 상태 기사의 고침/포털고침 인가 거부 (권한 게이팅)

- **Given** 어떤 기사의 생애주기 상태가 `DPS`이고, 현재 세션의 `role`이 `R`(또는 `Z`)이다
- **When** 그 세션이 해당 기사에 고침/포털고침(edit/portal-edit) 액션을 시도한다
- **Then** 시스템은 세션 `role`을 근거로 인가를 판정하여 액션을 **거부**하고, 어떤 생애주기 전이도 계산하기 전에 차단하며, 기사 상태를 변경하지 않는다 (REQ-AUTH-ROLE-002/003)
- **And** 동일 액션을 `role`이 `D`인 세션이 시도하면 인가가 허용된다 (REQ-AUTH-ROLE-002)
- 커버: REQ-AUTH-ROLE-002/003/004 — news.md 76-77 (DPS는 D만 고침/포털고침)

## 시나리오 4 — 사용자 관리 인가 + 소프트 삭제 (신규 인가)

- **Given** 현재 세션의 `role`이 `Z`(관리자)이다
- **When** 그 세션이 사용자 삭제(비활성화)를 요청한다
- **Then** 시스템은 요청을 허용하고, 대상 User 행을 **물리 삭제하지 않고** 상태 기반 비활성화로 처리한다 (REQ-AUTH-USRMGMT-001/003)
- **And** 비활성화된 사용자는 이후 로그인으로 새 세션을 수립할 수 없으나, DB의 User 행은 보존된다 (REQ-AUTH-USRMGMT-004)
- **And** `role`이 `R`/`D`인 세션이 동일 요청을 하면 거부되고 어떤 User 행도 변경되지 않는다 (REQ-AUTH-USRMGMT-002)
- 커버: REQ-AUTH-USRMGMT-001~004 — CLAUDE.md HARD 규칙(DB 미삭제)

## 엣지 케이스 (Edge Cases)

### EC-1 — 만료 세션으로 보호 자원 접근
- **Given** 세션이 만료되었다
- **When** 클라이언트가 만료된 세션 쿠키로 보호 자원(기사 작성/조회/함수)에 접근한다
- **Then** 시스템은 후속 요청을 미인증으로 취급하고, 오래된 세션 콘텐츠를 제공하지 않으며, 재인증을 요구한다
- 커버: REQ-AUTH-GUARD-003

### EC-2 — 권한 없는 사용자의 고침 시도 (세션 role 위조 방지)
- **Given** `role`이 `R`인 세션이 활성 상태다
- **When** 클라이언트가 요청 본문/헤더에 `role=D`를 위조하여 DPS 기사 고침을 시도한다
- **Then** 시스템은 acting `role`을 **검증된 서버 세션에서만** 도출하므로 클라이언트가 제공한 `role`을 무시하고, 실제 세션 role(`R`)로 인가를 판정하여 거부한다
- 커버: REQ-AUTH-ROLE-004, REQ-AUTH-ROLE-002

### EC-3 — 세션 쿠키 없는 보호 요청
- **Given** 세션 쿠키가 없거나 무효한 식별자다
- **When** 보호 자원에 접근한다
- **Then** 시스템은 미인증으로 거부하고 보호 자원을 제공하지 않는다 (서버가 SPEC-FRONTEND-UI-001 REQ-FE-APP-004의 권위 역할)
- 커버: REQ-AUTH-GUARD-002

### EC-4 — 로그아웃 후 재사용 시도
- **Given** 사용자가 로그아웃하여 세션이 무효화되고 쿠키가 만료되었다
- **When** 동일 세션 식별자로 보호 자원에 접근한다
- **Then** 시스템은 미인증으로 거부한다
- 커버: REQ-AUTH-SESS-004, REQ-AUTH-GUARD-002

## 품질 게이트 (Quality Gate Criteria)

- 인증 성공/실패 분기가 모두 테스트로 커버된다(시나리오 1, 2).
- 세션 수립·검증·무효화·만료 경로가 모두 검증된다(시나리오 1, EC-1, EC-4).
- R/D/Z 인가 매트릭스가 DPS 게이팅 포함하여 검증된다(시나리오 3, EC-2).
- 클라이언트 제공 `role` 위조가 무시됨이 검증된다(EC-2).
- 사용자 소프트 삭제가 물리 DELETE를 발행하지 않음이 검증된다(시나리오 4) — DB 행 보존 확인.
- 모든 응답에 비밀번호 해시가 포함되지 않음이 검증된다.

## Definition of Done

- [ ] 모듈 ①~⑤의 모든 REQ가 구현되고 대응 테스트 통과
- [ ] 시나리오 1~4 + EC-1~EC-4 모두 통과
- [ ] 세션 = 서버 측 세션 + 쿠키 (JWT 미사용) 확인
- [ ] 사용자 삭제 = 소프트 삭제, 물리 DELETE 부재 확인 (CLAUDE.md HARD 규칙)
- [ ] 인가 판정이 세션 `role`에만 근거함 확인 (클라이언트 값 불신)
- [ ] 비밀번호 해시 비노출 확인
- [ ] 기존 SPEC(BACKEND/FRONTEND/DB) 계약 회귀 없음
- [ ] TRUST 5 품질 게이트 통과 (Tested 85%+, Secured OWASP 정렬)
