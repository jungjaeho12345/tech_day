## 1. 라우팅 / 공통 사용자 정보 (REQ-FE-APP-002/003/004)

### AC-1.1 인증된 모든 페이지에 우측 상단 사용자 정보 표시
- **Given** 사용자가 로그인에 성공하여 인증 세션이 있고 `name`과 `role`이 가용한 상태에서
- **When** 로그인/작성/조회 중 임의 인증 페이지를 렌더링하면
- **Then** 우측 상단 영역에 로그인 사용자의 `name`과 `role`이 표시된다.

### AC-1.2 미인증 접근 차단
- **Given** 인증 세션이 없는 상태에서
- **When** 작성 페이지 또는 조회 페이지로 직접 진입을 시도하면
- **Then** 해당 페이지가 렌더링되지 않고 로그인 페이지로 라우팅된다.

## 2. 로그인 흐름 (REQ-FE-LOGIN-001~005)

### AC-2.1 로그인 성공 → 작성 페이지 이동
- **Given** 로그인 페이지에 아이디/암호 입력 폼이 있고 유효한 `userId`/`password`가 입력된 상태에서
- **When** 폼을 제출하고 백엔드가 인증 성공을 반환하면
- **Then** 작성 페이지로 이동하고, 공통 상단 사용자 정보 요소에 인증 사용자 `name`/`role`이 반영된다.

### AC-2.2 로그인 실패 → 에러 표시, 이동 없음
- **Given** 로그인 폼에 값이 입력된 상태에서
- **When** 폼을 제출하고 백엔드가 인증 실패를 반환하면
- **Then** 로그인 에러 메시지가 표시되고 로그인 페이지에 머문다(이동 없음).

### AC-2.3 비밀번호 마스킹
- **Given** 로그인 페이지가 렌더링된 상태에서
- **When** 암호 입력란에 문자를 입력하면
- **Then** 입력 문자가 마스킹되어 평문으로 표시되지 않는다.

## 3. 작성 페이지 레이아웃/탭/필드 (REQ-FE-WRITE-001~006, 015)

### AC-3.1 2영역 레이아웃 + 4탭 + 송고/보류 버튼
- **Given** 작성 페이지가 렌더링된 상태에서
- **When** 페이지를 확인하면
- **Then** 좌측 에디터 영역과 우측 메타데이터 영역이 구분되어 있고, 우측에 4탭(공통정보/이미지/영상/글기사)이 있으며, 4탭 위에 송고·보류 버튼이 있다.

### AC-3.1b 에디터 어댑터 추상화 [DP-F1]
- **Given** 좌측 에디터가 교체 가능한 어댑터(마크업 in/out 계약)로 래핑된 상태에서
- **When** 에디터에 본문을 작성하고 저장하면
- **Then** 어댑터가 마크업을 출력하고 `markupVersion`은 덮어쓰기되며(이력 UI 없음), 구체 에디터 라이브러리 교체 시 어댑터 계약을 유지하는 한 상위 화면/DTO 조립이 영향받지 않는다.

### AC-3.2 탭 전환
- **Given** 작성 페이지의 메타데이터 영역에서
- **When** 특정 탭을 선택하면
- **Then** 해당 탭 패널만 표시되고 나머지 탭 패널은 숨겨진다.

### AC-3.3 공통정보 탭 전체 필드
- **Given** 공통정보 탭이 활성화된 상태에서
- **When** 탭 패널을 확인하면
- **Then** 작성자·공동작성·내용·지역·속성·키워드·내부코멘트·외부코멘트·첨부파일·자료파일·엠바고 시간·2차 엠바고 시간 입력 필드가 모두 존재한다.

## 4. 검색 폴백 / 임베딩 (REQ-FE-WRITE-007~011) [DP-F3: 백엔드 프록시 경유]

### AC-4.1 이미지/영상 검색 — 백엔드 프록시 경유, YouTube 우선
- **Given** 이미지 또는 영상 탭이 활성화된 상태에서
- **When** 검색어로 검색을 실행하면
- **Then** 프런트엔드 Model이 백엔드 검색 프록시 인터페이스를 호출하고(클라이언트가 외부 API 키를 직접 보유하지 않는다), 프록시가 먼저 YouTube를 검색하여 그 결과가 표시된다.

### AC-4.2 YouTube 실패 시 Google 폴백 (서버 측)
- **Given** 이미지 또는 영상 검색을 실행한 상태에서
- **When** 프록시의 YouTube 검색이 실패(무결과 또는 오류)하면
- **Then** 백엔드 검색 프록시가 동일 검색어로 Google 검색을 수행하여 결과가 표시되며, 프런트엔드는 제공자와 무관하게 단일 검색 인터페이스로 결과를 받는다.

### AC-4.3 글기사 내부 검색 + 임베딩
- **Given** 글기사 탭이 활성화된 상태에서
- **When** 기사 제목/본문내용으로 검색하고 결과 항목을 선택하면
- **Then** 매칭된 내부 기사가 표시되고 선택 항목이 본문에 임베딩된다.

## 5. 송고/보류 + DTO (REQ-FE-WRITE-012~014)

### AC-5.1 송고 → 확인창 → DTO 조립·제출, 성공 시 상태 메시지 미표시 [DP-F5] (v0.3.0 개정)
- **Given** 에디터 본문과 4탭 입력이 채워진 상태에서
- **When** 송고 버튼을 누르면
- **Then** '송고하시겠습니까?' 확인창이 먼저 표시되고, 확인 시 UI는 액션 + (에디터 + 4탭 입력으로 조립된) 기사 DTO만 백엔드에 전송하며, 다음 상태를 클라이언트가 계산하지 않고, 요청이 성공하면 버튼 아래에 상태 메시지를 표시하지 않으며 작성 페이지가 초기화된다.

### AC-5.2 보류 액션 제출 (확인창 + DTO 저장 선행) (v0.3.0 개정)
- **Given** 현재 기사가 작성 페이지에 로드된 상태에서
- **When** 보류 버튼을 누르면
- **Then** '보류하시겠습니까?' 확인창이 먼저 표시되고, 확인 시 현재 기사 DTO가 저장된 뒤 보류 액션이 백엔드에 제출되며, 요청이 성공하면 버튼 아래에 상태 메시지를 표시하지 않는다.

### AC-5.3 KILL 액션 제출 (확인창 + DTO 저장 선행) (v0.3.0 신설)
- **Given** 현재 기사가 작성 페이지에 로드된 상태에서
- **When** KILL 버튼을 누르면
- **Then** 'KILL하시겠습니까?' 확인창이 먼저 표시되고, 확인 시 현재 기사 DTO가 저장된 뒤 KILL 액션이 백엔드에 제출되며, 요청이 성공하면 버튼 아래에 상태 메시지를 표시하지 않는다.

### AC-5.4 확인창 취소 시 미진행 (v0.3.0 신설)
- **Given** 작성 페이지에서 송고/보류/KILL 버튼을 누른 상태에서
- **When** 확인창에서 취소를 선택하면
- **Then** 저장(saveArticle)과 액션 제출(applyAction)이 모두 호출되지 않고 페이지 상태가 변하지 않는다.

## 6. 조회 페이지 실시간 + 상태바 (REQ-FE-VIEW-001~003) [DP-F2: 구독 인터페이스]

### AC-6.1 실시간 갱신 + 상태바 (구독 인터페이스 경유)
- **Given** 조회 페이지가 활성 메뉴로 표시된 상태에서
- **When** 구독 인터페이스가 활성 메뉴 필터에 해당하는 새/변경 기사 데이터를 보고하면
- **Then** 수동 새로고침 없이 기사 목록이 갱신되고, 우측 상단 실시간 상태바가 실시간 상태를 표시한다(전송 방식이 WebSocket인지 폴링인지와 무관하게 동일 동작).

## 7. 조회 페이지 4개 메뉴 (REQ-FE-VIEW-004~008)

### AC-7.1 부서별 작성 (v0.4.0 개정 — 부서 Select + DPS·RRH 제외)
- **Given** 부서별 작성 메뉴가 활성이고 부서 Select가 분리된 데이터-소스 인터페이스(User.department distinct)로 채워진 상태에서
- **When** 메뉴에 진입하면
- **Then** 로그인 사용자의 부서가 Select에 기본 선택되어 자동 조회되고, 해당 부서에서 작성된 기사 중 상태값이 DPS 또는 RRH가 **아닌** 기사만 표시된다(쿼리 필터 `{ department, statusNot: 'DPS,RRH' }`).

### AC-7.1b 부서별 작성 — 다른 부서 재조회 (v0.4.0 신설)
- **Given** 부서별 작성 메뉴가 활성인 상태에서
- **When** 부서 Select에서 다른 부서를 선택하고 조회 버튼을 누르면
- **Then** 선택한 부서의 기사 중 상태값이 DPS 또는 RRH가 아닌 기사만 표시된다.

### AC-7.2 부서별 송고 (드롭다운 + 조회 버튼) [DP-F4: 데이터-소스 인터페이스]
- **Given** 부서별 송고 메뉴가 활성이고 부서 드롭다운이 분리된 데이터-소스 인터페이스(예: User.department distinct)로 채워진 상태에서
- **When** 부서 드롭다운에서 부서를 선택하고 조회 버튼을 누르면
- **Then** 선택한 부서의 기사만 표시된다(조회 버튼을 누르기 전에는 해당 부서 결과가 채워지지 않는다).

### AC-7.3 개인별 수정 (v0.4.0 개정 — RDS·RRK만, v0.5.0 개정 — author 매칭 기준 명시)
- **Given** 개인별 수정 메뉴가 활성인 상태에서
- **When** 목록을 확인하면
- **Then** 현재 개인 사용자가 작성한 기사 중 상태값이 RDS 또는 RRK인 기사만 표시된다(쿼리 필터 `{ author, status: 'RDS,RRK' }` — v0.5.0: `author` 값은 기사에 저장된 작성자 표시 이름, 즉 로그인 사용자의 `user.name`과 매칭한다. 기사 author 컬럼에는 이름이 저장되므로 userId 매칭은 항상 0건이었다).

### AC-7.4 데스크 미송고 (v0.3.0 개정, v0.5.0 개정 — 기사상태 컬럼 추가)
- **Given** 데스크 미송고 메뉴가 활성인 상태에서
- **When** 목록을 확인하면
- **Then** 상태값이 RDS 또는 DDH인 기사만 표시되고, 각 기사 행은 기사아이디, 제목, 작성자, 수정자, 작성시간, 수정시간, 기사상태(status), LockYN 8개 컬럼을 표현한다.

### AC-7.5 전 메뉴 공통 8컬럼 (v0.4.0 신설, v0.5.0 개정 — 기사상태 컬럼 추가)
- **Given** 데스크 미송고, 부서별 작성, 부서별 송고, 개인별 수정 중 임의 메뉴가 활성이고 기사 목록이 표시된 상태에서
- **When** 기사 행을 확인하면
- **Then** 모든 메뉴에서 기사 행은 기사아이디, 제목, 작성자, 수정자, 작성시간, 수정시간, 기사상태(status — RDS/DPS/RRH/RRK/DDH/DDK 원시 값), LockYN 8개 컬럼을 표현한다(인라인 액션 버튼은 SPEC-NEWS-REVISE-007이 정의한 부서별 송고 DPS 행 고침/포털고침 진입점만 예외로 허용).

## 8. 권한 게이팅 (REQ-FE-VIEW-009/010)

### AC-8.1 DPS 기사 고침/포털고침 — D 권한만 사용 가능 (v0.4.0 개정 — 우클릭 메뉴)
- **Given** 조회 페이지에 상태값 DPS인 기사가 표시되고 현재 사용자 `role`이 `D`인 상태에서
- **When** 해당 기사 행에서 우클릭 컨텍스트 메뉴를 열면
- **Then** 고침(포털제외)/포털고침 메뉴 항목이 활성화되어 있고, 선택하면 해당 기사 편집 컨텍스트로 기사작성 페이지로 이동한다. (v0.4.0: 7컬럼 통일로 인라인 버튼은 제거 — 게이팅은 컨텍스트 메뉴에서 수행)

### AC-8.2 DPS 기사 — R/Z 권한은 비활성 (v0.4.0 개정 — 우클릭 메뉴)
- **Given** 조회 페이지에 상태값 DPS인 기사가 표시되고 현재 사용자 `role`이 `R` 또는 `Z`인 상태에서
- **When** 해당 기사 행에서 우클릭 컨텍스트 메뉴를 열면
- **Then** 고침(포털제외)/포털고침 메뉴 항목이 비활성화되어 액션을 제공하지 않는다. 상태값이 DPS가 아닌 기사에서는 역할과 무관하게 비활성이다.

## 엣지 케이스 (Edge Cases)

- **EC-1**: 로그인 응답에 비밀번호 해시가 포함되어도 UI는 이를 저장·표시하지 않는다(REQ-USR-LOGIN-004 정렬).
- **EC-2**: 백엔드 검색 프록시에서 YouTube·Google 검색이 모두 실패하면 프록시가 "결과 없음"을 반환하고 UI는 결과 없음 상태를 표시하며 앱이 크래시하지 않는다([DP-F3]).
- **EC-2b**: 백엔드 검색 프록시 호출 자체가 실패(네트워크/프록시 오류)하면 UI는 검색 오류 상태를 표시하고 다른 탭·작성 기능은 계속 동작한다([DP-F3]).
- **EC-3**: 구독 인터페이스의 실시간 연결이 끊기면([DP-F2] 전송 방식 무관) 상태바가 비-실시간/재연결 상태를 표시한다.
- **EC-4**: 부서별 송고에서 부서를 선택하지 않고 조회를 누르면 명확한 안내(또는 전체 미표시)로 처리하며 잘못된 부서 결과를 표시하지 않는다.
- **EC-5**: 송고/보류 제출이 백엔드에서 거부(유효하지 않은 전이, REQ-ART-LC-008)되면 상태 변경을 표시하지 않고 사용자에게 거부를 알린다.
