## 2. 개발 방식 비교 (Process)

### 2.1 BEFORE — 단발 구현 방식

- 흐름: news.md → 사람이 SPEC 수기 → expert-frontend/backend 1회 위임 → 사람이 jest/build 실행 → 종료
- 결함 수정: GitHub issue 기반 PR(#1·#3·#5·#9·#13·#17·#19·#21)로 사후 패치

### 2.2 AFTER — 하네스 엔지니어링

```
manager-spec → expert-frontend → evaluator-active → (PASS면 sync / FAIL면 loop / STAGNATION이면 escalation)
                                        ▲                       │
                                        └───────────────────────┘
                                   max 5 라운드 / pass_threshold 0.75 / floor 0.60(FROZEN)
```

- 메타 SPEC `SPEC-HARNESS-NEWS-001` 이 하네스 자체를 EARS REQ로 명세
- 인텐트 3개(`/news produce|plan|verify`)로 전체 파이프라인 노출 — thin command 라우팅
- Phase 1 Spec / Phase 3 Evaluate 는 skip 불가(FROZEN)
- SPEC-NEWS-REVISE 시리즈 **15종**을 동일 하네스로 반복 생산

## 9. 토큰 종류별 비용 구성 + 컨텍스트 재참조

### 9.1 컨텍스트 재참조(cache-read)란

LLM은 상태(기억)가 없어, **매 턴 전체 맥락(시스템 프롬프트 + CLAUDE.md + 규칙 + 누적 대화 + 읽은 파일)을 다시 입력받아야** 답할 수 있다. 이를 매번 정가로 보내면 비싸므로 프롬프트 캐싱을 쓴다:

- **cache-write(캐시 쓰기, $18.75/1M)** — 맥락을 처음 캐시에 올릴 때. 정가의 1.25배(1회성).
- **cache-read(캐시 읽기 = 컨텍스트 재참조, $1.5/1M)** — 캐시된 맥락을 다음 턴에서 다시 불러올 때. 정가의 **1/10**.

즉 "컨텍스트 재참조"는 **이전 턴들에서 쌓인 대화·파일 맥락을 다음 턴에 다시 모델에 먹이는 행위**다. 단가는 가장 싸지만 횟수·누적량이 압도적이라 총비용의 대부분을 차지한다. 하네스에서 폭증하는 이유: multi-agent가 **각자 전체 맥락을 로드**, GAN 루프가 같은 SPEC을 **최대 5라운드 반복 재참조**, 긴 세션일수록 누적 대화가 커진다.

### 9.2 토큰 종류별 비용 분해

| 토큰 종류 | BEFORE 비용 | (비중) | AFTER 비용 | (비중) | 배수 |

| 입력(정가) | $15.57 | 2.7% | $46.63 | 1.9% | 3.0배 |
| **출력** | **$112.16** | **19.8%** | **$269.90** | **11.2%** | **2.4배** |
| 캐시 쓰기 | $216.01 | 38.1% | $637.70 | 26.4% | 3.0배 |

## 10. 컨텍스트 재참조 비용 절감 방안

**메커니즘 자체는 불가피하지만(상태 없는 모델은 매 턴 맥락 재입력 필요), 총량은 설계로 결정된다.** 프롬프트 캐싱이 이미 재참조를 1/10로 깎고 있어 "최악(정가 10배)"은 회피된 상태이며, 추가 절감 레버는 다음과 같다:

1. **항상 실리는 맥락 줄이기** — CLAUDE.md/규칙/자동로드 파일은 매 턴 통째로 재참조된다. 이 프로젝트는 규칙 문서가 크다 → progressive disclosure·규칙 슬림화가 직접 효과.
2. **서브에이전트 스코프 축소** — 에이전트마다 전체 맥락을 새로 로드한다. 좁은 프롬프트 + 필요한 파일만 전달.
3. **GAN 라운드 수 억제** — 5라운드 반복은 같은 맥락을 5번 재참조. 불필요한 루프 축소가 직접 절감(품질 임계·escalation 튜닝).
4. **세션 분할/정리** — `/clear`·`/compact`로 누적 대화를 끊어 재참조 베이스 축소.
5. **타깃 읽기** — 파일 전체 대신 Grep/offset로 필요한 구간만 읽기.

요약: 이 프로젝트의 cache-read 6.5배 증가는 multi-agent×GAN 반복의 산물이므로, 위 1~3을 조이면 **품질 게이트를 유지하면서도** 재참조 총량을 상당 폭 줄일 수 있다.
