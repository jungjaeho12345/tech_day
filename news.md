# news.md
기사 작성기 기술 명세서이다.

### 서버
##기술명세서
- NodeJS를 사용한다.
- MVC Pattern으로 개발한다.
- DB는 SQLite를 사용한다.

## 기사 함수
- 기사를 입력/수정/조회 할 수 있는 함수를 생성한다.
- 기사를 조회하는 함수에서는 배부시간을 조건으로 기사를 조회할 수 있다.
- 기사를 조회하는 함수에서는 작성시간을 조건으로 기사를 조회할 수 있다.
- 기사를 조회하는 함수에서는 기사 아이디를 조건으로 기사를 조회할 수 있다.
- 기사를 조회하는 함수에서는 작성자를 조건으로 기사를 조회할 수 있다.
- 기사를 조회하는 함수에서는 송고자를 조건으로 기사를 조회할 수 있다.
- 기사를 수정하는 함수에서는 기사 아이디를 조건으로 기사 상태값을 수정할 수 있다.
- 기사를 삭제하는 함수에서는 기사 아이디를 조건으로 기사 상태값을 수정할 수 있다.

## 사용자 함수
- 사용자를 입력/수정/삭제/조회 할 수 있는 함수를 생성한다.
- 사용자는 조회하는 함수에서 아이디와 비밀번호를 조건으로 전달된 값이 DB에 있는 정보와 같으면 로그인 할 수 있다.
- 로그인 후에는 기사 작성 기능과 기사 목록기능이 있는 페이지로 이동한다.

## 워크플로우
- 클라이언트에서 요청받은 데이터를 사용하여 기사 생애주기를 거쳐 DB에 적용된다.

# 기사 작성페이지
- 기사 작성페이지는 에디터와 우측에 기사 메타데이터를 입력하는 레이아웃으로 구분한다.
- 기사 작성페이지에 좌측에 에디터와 우측에 기사 메타데이터는 60:40 비율로 설정해줘
- 기사 메타데이터를 입력하는 레이아웃에서 탭이 4개가 있는데 공통정보/이미지/영상/글기사가 있다.
- 4개 탭 위에 송고 보류 버튼이 있다.
- 공통정보에는 작성자, 공동작성, 내용, 지역, 속성, 키워드, 내부코멘트, 외부코멘트, 첨부파일, 자료파일을 입력받고, 엠바고, 2차 엠바고 시간을 입력받는다.
- 이미지와 영상, 글기사는 본문에 임베딩으로 사용할 데이터를 조회한다.
- 이미지 탭은 Google 이미지 검색을, 영상 탭은 YouTube 검색을 사용한다.
- 이미지/영상 검색은 서버 프록시 API를 통해 호출하며, 검색 API 키는 서버 환경변수로 설정한다. 외부 검색이 실패하면 오류 대신 빈 결과를 보여준다.
- 글기사는 내부 기사DB에서 제목, 본문내용을 검색하여 결과를 보여준다.
- 기사 작성페이지에서 기사 조회페이지로 이동했다가 돌아와도 작성 중이던 내용(제목/본문/임베드/공통정보)은 유지된다. 단, 송고/보류/KILL 성공 후의 작성페이지 초기화는 그대로 적용되고, 편집 진입(기사아이디 로드)은 보존된 내용보다 우선한다.
- 기사 작성 에디터는 탭(Tab)으로 여러 개 열 수 있다. ＋ 버튼으로 새 작성 탭을 추가하며, 각 탭의 작성 내용은 서로 독립적으로 유지된다. 탭을 닫으면 그 탭의 작성 내용은 폐기되고, 마지막 탭을 닫으면 빈 새 기사 탭 1개가 유지된다.
- 기사 조회페이지에서 편집 또는 고침/포털고침 기능으로 기사 작성 에디터를 로딩할 때는 해당 기사를 새로운 탭으로 만들어 보여준다. 이미 열려 있는 기사를 다시 열면 새 탭을 만들지 않고 그 탭을 활성화한다 (동시 편집 잠금 규칙과 정합).
- 편집 탭에서 송고/보류/KILL이 성공하면 그 탭은 빈 새 기사 탭으로 전환된다 (편집 잠금 해제).
- 편집 탭이 열려 있는 동안에는 해당 기사의 편집 잠금(lockYN='Y')이 유지된다. 기사 조회페이지로 이동해도 잠금은 해제되지 않는다. 잠금 해제 시점은 편집 탭 닫기(×), 송고/보류/KILL 성공, 로그아웃/세션 만료, 브라우저(탭) 닫힘이다.
- 탭 목록과 탭별 작성 내용은 sessionStorage에 저장되어 페이지 이동 후에도 유지된다.
- 탭을 전환하면 주소창이 활성 탭에 맞게 갱신된다 (편집 탭이면 기사아이디 쿼리 표시, 새 기사 탭이면 제거). 브라우저 탭 제목은 활성 편집 탭의 기사아이디로 표시한다.
- 다른 창에서 편집 중인 기사의 잠금이 강제 해제(Lock해제)되면 그 기사의 편집 탭은 자동으로 닫힌다.

# 기사 작성 페이지 내 버튼
- Z권한은 송고/보류/KILL 버튼이 보이고 사용할 수 있다.
- 송고/보류/KILL 버튼은 권한 R, D 그리고 RDS, DDH 기사일 떄 표현한다.
- 송고/보류 버튼은 권한 R, D 그리고 DPS 기사가 포털고침 또는 고침버튼을 
눌렀을 때 표현한다.
- KILL 버튼은 권한 R 그리고 RDS 기사일 때 표현한다.
- 기사아이디가 생성되지 않은 기사(저장 전 신규 작성)를 기사작성 페이지에서 편집할 때는 화면에 KILL 버튼을 표시하지 않는다.
- 데스크 보류(DDH) 기사를 편집할 때는 송고와 KILL 버튼을 권한 D, Z에게 표현한다. 보류 버튼은 표현하지 않는다(이미 보류 상태). 권한 R에게는 어떤 버튼도 표현하지 않는다.
- 송고/보류 버튼을 누르면 articleUpdate API를 호출해 기사 DB의 Contents 테이블내에서 상태값을 기사 생애주기 조건에 맞게 변경한다.
- 송고는 본문에 "(끝)" 표시가 있어야 한다. 없으면 ALERT를 띄우고 송고를 차단한다. 보류/KILL은 "(끝)" 없이도 진행된다.

# 기사 조회페이지
- 기사 조회페이지는 실시간이다.
- 실시간은 SSE 스트림으로 구현한다. 기사 생성/수정/상태변경/잠금변경 이벤트를 수신하면 목록을 갱신하고, 연결이 끊기면 자동으로 재연결한다.
- 실시간 상태를 보여주는 상태바 우측 상단에 있다.
- 메뉴가 4개 있는데 데스크 미송고(Default), 부서별 작성, 부서별 송고, 개인별 수정 메뉴가 있다.
- 부서별 작성페이지는 부서별로 조회할 수 있는 Select 메뉴와 조회버튼이 있고, 진입 시 로그인 사용자의 부서가 기본 선택되어 자동 조회된다.
- 부서별 작성페이지는 해당 부서에서 작성된 기사 중 상태값이 DPS와 RRH가 아닌 기사만 조회한다.
- 부서별 송고페이지는 부서별로 송고할 수 있는 드롭다운 메뉴와 조회버튼을 통해 조회할 수 있다.
- 부서 선택은 '전체' 토글이 있는 체크박스 멀티셀렉트 드롭다운으로, 여러 부서를 한 번에 선택해 조회할 수 있다.
- 부서별 송고페이지는 DPS기사만 조회 할 수 있다.
- 개인별 수정페이지는 로그인한 계정이 작성한 기사 중 상태값이 RDS, RRK인 기사만 조회 할 수 있다.
- 부서별 작성 페이지, 부서별 송고 페이지, 개인별 수정 페이지에서 우클릭 하면 상세보기/이력보기/송고이력보기/ 본문복사/제목만복사/번역/매핑/후속기사작성/계속기사작성/고침(포털제외)/포털고침/삭제요청/재송 기능이 있다.
- 우클릭 메뉴의 고침(포털제외)/포털고침은 상태값이 DPS인 기사에서 D 권한 사용자에게만 활성화되며, 누르면 기사작성 페이지로 이동하여 편집할 수 있다.
- 부서별 송고 페이지의 우클릭 메뉴에는 편집 항목이 추가로 있다.
- 우클릭 메뉴의 이력보기/송고이력보기/번역/매핑/후속기사작성/계속기사작성/삭제요청/재송 항목은 메뉴에 표시만 되고 아직 동작하지 않는다 (비활성).
- 잠긴(LockYN='Y') 기사의 우클릭 메뉴에는 Lock해제 항목이 나타난다. 권한 D, Z만 사용할 수 있고 '해제하시겠습니까?' 확인 후 해제하며, 권한 R에게는 비활성으로 보인다.
- 데스크 미송고 페이지는 상태값이 RDS, DDH인 기사만 나열한다.
- 데스크 미송고 목록의 컬럼은 기사아이디, 제목, 작성자, 수정자, 작성시간, 수정시간, 기사상태, LockYN만 표현한다.
- 데스크 미송고 페이지에서 우클릭 하면 편집/상세보기/이력보기/본문복사/제목만목사 가 있다.
- 데스크 미송고 페이지의 기사는 편집 가능하며 우클릭 편집을 누르면 기사작성 페이지로 이동하여 편집 할 수 있다.
- 조회페이지에 기사는 시간 내림차순이고, 4개 메뉴(데스크 미송고, 부서별 작성, 부서별 송고, 개인별 수정) 모두 기사 목록의 컬럼은 기사아이디, 제목, 작성자, 수정자, 작성시간, 수정시간, 기사상태, LockYN만 표현한다.
- 기사를 클릭하면 새로운 창에서 기사의 제목, 내용, 공통정보 내용을 볼 수 있다.
- 기사는 10개씩 보여주며 페이징 처리 해줘
- 기사 목록 헤더에서 우클릭하면 컬럼 표시/숨김과 컬럼 간격을 설정하는 모달이 열리고, 설정은 메뉴별로 저장된다.
- 작성시간/수정시간 컬럼은 YYYY-MM-DD HH:mm 형식으로 가운데 정렬해 표시한다.
- 상태값 배지 색은 RDS는 회색, DPS는 레드, 보류(RRH/DDH)는 앰버, KILL(RRK/DDK)은 슬레이트로 구분한다.

# 상세보기
- 상세보기를 누르면 새로운 창에서 상단에는 공통정보 작성자, 공동작성, 내용, 지역, 속성, 키워드, 내부코멘트, 외부코멘트, 첨부파일, 자료파일, 엠바고 시간, 2차 엠바고 시간을 세로가 아닌 가로로 나열해서 보여주고, 하단에는 기사 본문을 하나의 영역에서 보여준다. 본문 첫 줄이 곧 제목이므로 별도의 제목 요소는 두지 않는다(중복 제거).
- 상세보기 새 창은 720×800 크기로 열리고, 창 제목은 기사 제목이다 (제목이 없으면 '(제목 없음)').
- 공통정보의 빈 필드는 '—'로 표시한다.
- 본문은 저장된 본문 구조(텍스트/임베드 블록) 순서대로 보여주며, 모든 내용은 HTML 이스케이프되어 스크립트가 실행되지 않는다.

# 사용자 정보
- 기사 작성페이지와, 기사 조회페이지 우측 상단에는 USER정보에 있는 유저아이디, 부서, 권한을 보여준다.
- 사용자 정보는 '유저아이디 · 부서 · (권한)' 형식으로 표시한다.
- 로그아웃 버튼이 있고, 로그아웃 버튼을 누르면 session을 종료하고 로그인 페이지로 redirect한다.

## 세션 정책
- 로그인한 사용자가 1시간 동안 동작이 없으면 세션이 만료된다. 동작이 있으면 만료 시점이 갱신되어 세션이 유지된다 (sliding expiration).
- 로그아웃 버튼을 누르기 전까지는 (활동이 있는 한) 세션을 유지한다.
- 새로고침(F5)을 눌러도 세션이 끊어지면 안 되고 유지한다. (탭/브라우저 닫힘 시에는 세션 종료 — lockYN 규칙과 정합)
- 세션 아이디는 서버가 발급하는 무작위 토큰이며 권한 정보를 담지 않는다. 로그인 성공 시 기존 세션을 무효화하고 새 세션 아이디를 발급한다.
- 세션 만료 시점 갱신(sliding)은 인증된 모든 요청마다 이루어진다.
- 클라이언트는 사용자 정보와 세션 아이디를 sessionStorage에 저장하고, 새로고침 시 서버 확인을 거쳐 복원한다. 복원이 끝나기 전에는 로그인 페이지로 이동시키지 않는다.

## 편집 잠금 (lockYN)
- 기사 편집을 시작하면 해당 기사는 lockYN='Y'로 잠긴다. 잠금은 잠근 사용자, 세션, 잠금 시각으로 식별한다.
- 한 기사의 편집은 한 페이지 한 세션 한정이다. 같은 사용자라도 다른 페이지(다른 세션)에서는 잠긴 기사를 편집할 수 없다.
- 잠금 후 30분 동안 갱신이 없으면 만료된 잠금으로 보고, 다음 편집 시도자가 잠금을 가져갈 수 있다.
- 잠금 획득에 실패해도 누가 잠갔는지는 알려주지 않는다.
- 브라우저(탭)가 닫힐 때는 해제 요청을 보내 잠금을 해제한다.
- 강제 해제(Lock해제)는 권한 D, Z만 할 수 있다.

## 로그인 워크플로우
- 로그인 페이지에서 아이디와 암호를 서버에 전달해 DB에 USER 테이블에 있는 아이디와 암호를 비교하여 똑같으면 기사 작성페이지로 넘어간다. 틀리면 ALERT창을 띄운다.
- 로그인 페이지에서 아이디와 암호를 서버에 전달해 DB에 USER 테이블에 있는 아이디와 암호를 비교하여 똑같지 않으면  실패한 이유와 함께 ALERT 메시지를 띄운다.
- 로그인 성공한 사용자는 송고 버튼을 눌러 기사DTO를 서버에 전달해 기사를 DB에 적용한다.
- 로그인은 15분 동안 같은 IP에서 10회까지만 시도할 수 있다.
- 비밀번호는 해시로 저장하며, 어떤 응답에도 비밀번호를 포함하지 않는다.
- 비활성 처리된(active='N') 사용자는 로그인할 수 없다.
- 로그인 실패 시 원인과 관계없이 같은 시간이 걸리도록 처리한다.
- 아이디/암호 입력칸에는 '아이디를 입력하세요'/'암호를 입력하세요' 안내 문구를 표시한다.

## 기사작성 워크플로우
- 송고/보류/KILL 버튼을 누르면 먼저 '송고하시겠습니까?'/'보류하시겠습니까?'/'KILL하시겠습니까?' 확인창을 띄우고, 확인했을 때만 진행한다. 취소하면 아무것도 저장/전송하지 않는다.
- 송고/보류/KILL 버튼을 누르면 서버로 DB Article정보와 Contents정보가 전달되 DB Article 테이블과 Contents테이블에 저장된다
- 송고/보류/KILL 버튼을 누른 후 기사 작성페이지는 초기화 된다.
- 송고/보류/KILL 요청이 성공하면 버튼 아래에 상태 메시지를 표시하지 않는다.
- 송고/보류 버튼을 누를때 제목이 없으면 제목없어서 송고/보류 실패 ALERT를 띄운다.
- 편집 중 기사 저장(수정)은 편집 잠금을 보유한 세션만 할 수 있다.

## SQLite Store Procedure 명세서
- 기사 아이디를 생성해주는 SP를 만든다.
- 기사이이디 규칙은 'AKR' + YYYYMMDD + 난수 9자리로 생성한다.
- 생성된 기사 아이디가 이미 존재하면 난수 부분을 다시 생성하여 중복되지 않게 한다.

## 기사 에디터
 - 에디터 본문 영역 위에 '본문' 라벨 텍스트는 표시하지 않는다 (접근성 aria-label은 유지).
 - 첫째줄은 제목이다.
 - 둘째줄부터 5째줄까지 부제목, 단 개행라인이 2번이상 있으면 2번째부터는 본문이다.
 - 에디터는 /이미지/영상/글기사의 검색 결과에서의 데이터를 본문 커서 위치에 임베딩 할 수 있다. 엠베딩 후 결과는 유지한다.
 - 클립보드에서 복사하여 붙여넣기한 이미지/유투브 크기는 에디터크기가 100%이고 가로*세로=17%*17%이다. (기존 10%*10%에 1.7배 적용. 사진/영상 figure 폭도 1.7배=612px, 기사 참조 카드는 480px 유지)
 - 제목은 파란색, 부제목은 빨가색, 본문은 검정색이다.
 - Alt+Y를 누르면 "(끝)"을 임베드 뒤 최종 블록으로 삽입하고(최종 시각 순서: 본문 텍스트 → 임베드 → "(끝)") 색상은 골드색으로 보여준다.
 - Alt+Y를 누를시 이미 "(끝)"이 있다면 삽입하지 않는다.
 - Ctrl+d를 누르면 해당라인을 제거한다.
 - Backspace/Delete/Ctrl+D로 라인을 삭제할 때 그 라인의 임베드도 함께 삭제된다 (임베드는 한 번에 한 개씩).
 - 각 임베드에는 × 삭제 버튼이 있어 클릭으로 삭제할 수 있다.
 - 본문은 텍스트/임베드 블록 구조(markupVersion)로 저장되며, 편집-저장-불러오기를 반복해도 블록 순서가 보존된다. 이전 방식(일반 텍스트)으로 저장된 본문도 불러올 수 있다.
 - 한글 입력(IME) 조합 중에는 색상을 다시 칠하지 않는다. 색상 적용은 조합 완료, 포커스 이탈, 기사 불러오기 시점에 이루어진다.

## 사용자 권한
- 기자 리포터 R, 국기사 데스크 D, 관리자 Z

## 기사 제어 권한
- 기사는 사용자권한 R, D, Z 편집 할 수 있다.
- 기사의 상태값이 DPS일 때는 D 권한 사용자만 고침/포털고침 메뉴를 사용할 수 있다.
- Z권한은 기사 작성 및 데스크 미송고 편집 권한이 있다.

## 기사 편집 기능
- 편집 버튼 누를 시, 기사 작성 페이지내에 에디터에 제목과 본문내용 불러오며, 공통정보 작성자, 공동작성, 내용, 지역, 속성, 키워드, 내부코멘트, 외부코멘트, 첨부파일, 자료파일, 엠바고 시간, 2차 엠바고 시간 모두 불러온다.
- 데스크 미송고에서 편집 버튼 사용시 또는 부서별 송고에서 편집, 고침, 포털고침 사용시 기사 작성페이지로 포워딩되어 기사의 정보들(ContentsVO.md)이 매핑되어 보여준다.
- 매핑 시 제목/본문내용/작성자/엠바고 시간/2차 엠바고 시간은 기존 입력란에 채워지고, 나머지 ContentsVO 필드(기사아이디, 수정자, 송고자, 부서, 부서코드, 작성시간, 편집시간, 송고시간)는 작성 페이지에 읽기전용으로 표시한다.
- 부서별 송고 페이지의 DPS 행에 있는 고침/포털고침 버튼도 동일하게 기사 작성페이지로 포워딩한다 (D 권한만 사용 가능).
- 고침/포털고침으로 진입하는 것은 단순 편집 진입이며 기사 상태값 전이를 일으키지 않는다.

## 기사 생애주기
- 기사가 최초 작성시에는 상태값이 RDS이다.
- 작성 페이지에서 신규 기사를 최초 송고하면 권한과 무관하게 상태값 전이 없이 RDS로 저장된다 (데스크 미송고 목록에서 검수 대기). 아래 권한별 전이 규칙은 기존 기사(편집 컨텍스트)의 송고/보류/KILL에 적용된다. (2026-06-07 결정)
- 권한 R 사용자가 RDS 기사를 송고시에는 RDS가 된다.
- 권한 R 사용자가 RDS 기사를 보류시에는 RRH가 된다.
- 권한 R 사용자가 RDS 기사를 KILL 하면 RRK가 된다.
- 권한 D 사용자가 RDS 기사를 송고시에는 DPS가 된다.
- 권한 D 사용자가 RDS 기사를 보류시에는 DDH가 된다.
- 권한 D 사용자가 RDS 기사를 KILL 시에는 DDK가 된다.
- 권한 Z 사용자는 권한 D와 동일하게 전이한다 (RDS 기사를 송고시 DPS, 보류시 DDH, KILL시 DDK가 된다).
- DPS 기사를 송고하면 DPS가 유지되고(재송고), 보류하면 DDH가 된다. DPS 기사는 바로 KILL 할 수 없다 (보류 후 KILL 경로만 가능하다).
- DDH 기사를 권한 D, Z 사용자가 송고하면 DPS, KILL 하면 DDK가 된다. 권한 R 사용자는 DDH 기사에 어떤 액션도 할 수 없다.
- 정의된 전이 외의 (상태, 권한, 액션) 조합은 모두 거부된다.

## 기사 DB 명세서
- @schema.md를 참조한다.
- Contents 테이블에는 편집 잠금 컬럼(lockYN, 잠근 사용자/세션/시각)과 공통정보 컬럼(공동작성, 지역, 속성, 키워드, 내부코멘트, 외부코멘트, 첨부파일, 자료파일)이 있다.
- User 테이블에는 active 컬럼이 있어 'N'이면 로그인할 수 없는 비활성 사용자다.
- 스키마 변경은 기존 데이터를 삭제하지 않고 컬럼을 추가하는 방식(멱등 마이그레이션)으로만 적용한다. 예전 DB의 컬럼 표기(대소문자)와 비어 있는 부서 값은 서버가 자동으로 보정한다.
- 기사 저장 시 부서가 비어 있으면 로그인 사용자의 부서가 자동 입력된다.
- 기사 조회는 부서 다중 선택과 특정 상태 제외 조건을 지원한다.
- Article 테이블과 Contents 테이블을 함께 수정할 때는 트랜잭션으로 처리한다.

## 보안
- 서버는 보안 헤더(CSP 등)를 적용하고, 오류 응답에 내부 스택 정보를 노출하지 않는다.
- CORS는 개발 클라이언트(localhost:5173)만 허용한다.
- 사용자 생성/수정/삭제는 권한 Z만 할 수 있다.
- 정의되지 않은 권한의 요청은 거부한다.
